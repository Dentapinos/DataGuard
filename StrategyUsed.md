# Стратегии восстановления в Data Guard

## Обзор

В приложении Data Guard реализовано **7 стратегий восстановления**, каждая из которых предназначена для конкретных сценариев эксплуатации баз данных. Все стратегии реализуют паттерн **Strategy** и наследуются от `AbstractRestoreStrategy`.

---

## Стратегии восстановления

### 1. **STRICT** — Строгое восстановление

**Ключевые характеристики:**
- Полная проверка соответствия схемы и данных
- Требует полного совпадения колонок в таблицах
- Прерывает выполнение при любом расхождении

**Политики по умолчанию:**

| Политика | Значение | Описание |
|----------|----------|----------|
| SchemaPolicy | `STRICT_SCHEMA` | Строгая проверка схемы: схема должна полностью совпадать |
| RowConflictPolicy | `FAIL_ON_CONFLICT` | Прерывание при конфликте уникальных ключей |
| ForeignKeyPolicy | `ENFORCE_ALL` | Полная проверка внешних ключей |
| ErrorPolicy | `FAIL_FAST` | Немедленное прерывание при ошибке |

**Когда использовать:****
- ✅ Тестовые и стейджинговые среды
- ✅ Восстановление после катастрофического сбоя
- ✅ Когда нужна гарантия полной идентичности с бэкапом
- ✅ Процедуры аудита и валидации

**Особенности:** Выбрасывает исключение при любом расхождении — таблица/столбец отсутствует, дублируется PK, нарушается FK-связь.

---

### 2. **SAFE_MERGE** — Безопасное слияние

**Ключевые характеристики:**
- Гибкое восстановление с пропуском конфликтов
- Допускает небольшие расхождения в схеме (лишние/отсутствующие колонки)
- Пропускает дубликаты строк без ошибки

**Политики по умолчанию:**

| Политика | Значение | Описание |
|----------|----------|----------|
| SchemaPolicy | `RELAXED_SCHEMA` | Упрощённая проверка: допустимы расхождения в схеме |
| RowConflictPolicy | `SKIP_ON_CONFLICT` | Пропуск конфликтующих строк |
| ForeignKeyPolicy | `SKIP_VIOLATIONS` | Пропуск нарушений FK |
| ErrorPolicy | `LOG_AND_CONTINUE` | Логирование ошибок с продолжением |

**Когда использовать:**
- ✅ Постепенное обновление данных
- ✅ Восстановление в среду с частично изменённой схемой
- ✅ Тестирование новых версий бэкапа
- ✅ Когда допустимо частичное восстановление

**Особенности:** Таблицы, отсутствующие в целевой БД, будут пропущены (не вызовут ошибку).

---

### 3. **FORCE_REPLACE** — Агрессивная замена

**Ключевые характеристики:**
- Данные из бэкапа считаются «истиной» и полностью заменяют текущие данные
- Временно отключает проверку внешних ключей
- Обновляет существующие строки и добавляет новые

**Политики по умолчанию:**

| Политика | Значение | Описание |
|----------|----------|----------|
| SchemaPolicy | `RELAXED_SCHEMA` | Допустимы расхождения в схеме |
| RowConflictPolicy | `OVERWRITE_ON_CONFLICT` | Перезапись существующих строк |
| ForeignKeyPolicy | `TEMP_DISABLE` | Временное отключение FK |
| ErrorPolicy | `LOG_AND_CONTINUE` | Логирование ошибок с продолжением |

**Когда использовать:**
- ✅ Полное восстановление базы данных
- ✅ Тестовые среды, где текущие данные не важны
- ✅ Восстановление после полного сбоя
- ✅ Миграции, когда нужно «перезаписать» всё

**Особенности:** Использует `ON DUPLICATE KEY UPDATE` — count=1 означает INSERT, count=2 — UPDATE.

---

### 4. **APPEND_ONLY** — Только добавление

**Ключевые характеристики:**
- Новые строки добавляются, существующие не меняются и не удаляются
- Использует `INSERT IGNORE` для пропуска дубликатов
- Гарантирует сохранение всех текущих данных

**Политики по умолчанию:**

| Политика | Значение | Описание |
|----------|----------|----------|
| SchemaPolicy | `RELAXED_SCHEMA` | Допустимы расхождения в схеме |
| RowConflictPolicy | `SKIP_ON_CONFLICT` | Пропуск дубликатов |
| ForeignKeyPolicy | `SKIP_VIOLATIONS` | Пропуск нарушений FK |
| ErrorPolicy | `LOG_AND_CONTINUE` | Логирование ошибок с продолжением |

**Когда использовать:**
- ✅ Добавление новых записей в существующие таблицы
- ✅ Логирование событий и событийных данных
- ✅ Пополнение справочников
- ✅ Когда нельзя менять существующие данные (например, финансовые записи)

**Особенности:** Игнорирует строки с дублирующимися PK/UNIQUE, добавляет только новые.

---

### 5. **UPSERT_ALL** — Полная операция UPSERT

**Ключевые характеристики:**
- Для всех строк выполняется "upsert" (INSERT ... ON DUPLICATE KEY UPDATE)
- Существующие записи обновляются, отсутствующие — добавляются
- Не удаляет данные, отсутствующие в бэкапе

**Политики по умолчанию:**

| Политика | Значение | Описание |
|----------|----------|----------|
| SchemaPolicy | `RELAXED_SCHEMA` | Допустимы расхождения в схеме |
| RowConflictPolicy | `OVERWRITE_ON_CONFLICT` | UPSERT-операции для всех строк |
| ForeignKeyPolicy | `TEMP_DISABLE` | Временное отключение FK |
| ErrorPolicy | `LOG_AND_CONTINUE` | Логирование ошибок с продолжением |

**Когда использовать:**
- ✅ Синхронизация данных между БД
- ✅ Обновление справочников с сохранением структуры
- ✅ Восстановление в среду с изменённой схемой (для новых колонок)
- ✅ Сценарии, где нужно обновить существующие и добавить новые записи

**Особенности:** Полная семантика upsert — count=1 (INSERT), count=2 (UPDATE).

---

### 6. **DRY_RUN** — Сухой запуск (симуляция)

**Ключевые характеристики:**
- Не выполняет никаких записей в БД
- Проводит все проверки на наличие таблиц/колонок
- Эмулирует статистику восстановления

**Политики по умолчанию (жёстко заданы):**

| Политика | Значение | Описание |
|----------|----------|----------|
| SchemaPolicy | `RELAXED_SCHEMA` | Допустимы расхождения в схеме |
| RowConflictPolicy | `SKIP_ON_CONFLICT` | Пропуск конфликтов |
| ForeignKeyPolicy | `SKIP_VIOLATIONS` | Пропуск нарушений FK |
| ErrorPolicy | `LOG_AND_CONTINUE` | Логирование ошибок с продолжением |

**Когда использовать:**
- ✅ Тестирование стратегии перед реальным восстановлением
- ✅ Проверка совместимости бэкапа с текущей БД
- ✅ Предсказание поведения восстановления
- ✅ Обучение и демонстрация функциональности

**Особенности:** Полностью имитирует процесс, но без реальных SQL-вызовов.

---

### 7. **SAFE_SCHEMA_CHECK** — Только проверка схемы

**Ключевые характеристики:**
- Проверяет только схему (наличие таблиц и колонок)
- Не пишет никаких данных
- Используется для валидации перед восстановлением

**Политики по умолчанию (жёстко заданы):**

| Политика | Значение | Описание |
|----------|----------|----------|
| SchemaPolicy | `RELAXED_SCHEMA` | Допустимы расхождения в схеме |
| RowConflictPolicy | `SKIP_ON_CONFLICT` | Пропуск конфликтов |
| ForeignKeyPolicy | `SKIP_VIOLATIONS` | Пропуск нарушений FK |
| ErrorPolicy | `LOG_AND_CONTINUE` | Логирование ошибок с продолжением |

**Когда использовать:**
- ✅ Предварительная валидация бэкапа
- ✅ Проверка совместимости перед восстановлением
- ✅ Проверка целостности схемы
- ✅ Аудит структуры БД перед миграцией

**Особенности:** Не выполняет импорт данных — только проверяет структуру.

---

## Рекомендации по выбору стратегии

| Сценарий | Рекомендуемая стратегия | Обоснование |
|----------|------------------------|-------------|
| **Восстановление после аварии в продакшене** | `FORCE_REPLACE` | Быстрое полное восстановление |
| **Восстановление после аварии с гарантией идентичности** | `STRICT` | Гарантия полной идентичности с бэкапом |
| **Обновление данных в тестовой среде** | `DRY_RUN` → `FORCE_REPLACE` | Сначала симуляция, затем реальное восстановление |
| **Добавление новых записей без изменения старых** | `APPEND_ONLY` | Сохранение всех существующих данных |
| **Синхронизация данных между БД** | `UPSERT_ALL` | UPSERT-операции для всех строк |
| **Валидация перед восстановлением** | `SAFE_SCHEMA_CHECK` | Проверка совместимости без изменений |
| **Тестирование новых бэкапов** | `DRY_RUN` или `SAFE_MERGE` | Безопасное тестирование |
| **Аудит и восстановление в стейджинге** | `STRICT` | Гарантия идентичности |
| **Постепенное обновление с пропуском конфликтов** | `SAFE_MERGE` | Гибкое восстановление |

## Примеры использования через REST API

### 1. Строгое восстановление
```bash
curl -X POST "http://localhost:8080/api/restore?tier=DAILY" \
  -H "Content-Type: application/json" \
  -d '{
    "backupName": "backup-bm-2026-06-14T10-00-00.zip",
    "targetDatabase": "bm",
    "mode": "STRICT",
    "tables": null
  }'
```

### 2. Добавление новых записей
```bash
curl -X POST "http://localhost:8080/api/restore?tier=DAILY" \
  -H "Content-Type: application/json" \
  -d '{
    "backupName": "backup-bm-2026-06-14T10-00-00.zip",
    "targetDatabase": "bm",
    "mode": "APPEND_ONLY",
    "tables": null
  }'
```

### 3. Проверка совместимости (DRY_RUN)
```bash
curl -X POST "http://localhost:8080/api/restore?tier=DAILY" \
  -H "Content-Type: application/json" \
  -d '{
    "backupName": "backup-bm-2026-06-14T10-00-00.zip",
    "targetDatabase": "bm",
    "mode": "DRY_RUN",
    "tables": null
  }'
```

### 4. Анализ схемы
```bash
curl -X POST "http://localhost:8080/api/restore/analyze-schema?database=bm&tier=DAILY" \
  -H "Content-Type: application/json" \
  -d '{
    "backupName": "backup-bm-2026-06-14T10-00-00.zip",
    "targetDatabase": "bm"
  }'
```

### 5. Полное восстановление (FORCE_REPLACE)
```bash
curl -X POST "http://localhost:8080/api/restore?tier=WEEKLY" \
  -H "Content-Type: application/json" \
  -d '{
    "backupName": "backup-bm-2026-06-15T02-00-00.zip",
    "targetDatabase": "bm",
    "mode": "FORCE_REPLACE",
    "tables": null
  }'
```

### 6. UPSERT всех строк
```bash
curl -X POST "http://localhost:8080/api/restore?tier=DAILY" \
  -H "Content-Type: application/json" \
  -d '{
    "backupName": "backup-bm-2026-06-22T01-00-00.zip",
    "targetDatabase": "bm",
    "mode": "UPSERT_ALL",
    "tables": ["users", "products"]
  }'
```

### 7. Безопасное слияние (только новые данные)
```bash
curl -X POST "http://localhost:8080/api/restore?tier=DAILY" \
  -H "Content-Type: application/json" \
  -d '{
    "backupName": "backup-bm-2026-06-22T01-00-00.zip",
    "targetDatabase": "bm",
    "mode": "SAFE_MERGE",
    "tables": null
  }'
```

## Сравнительная таблица стратегий

| Стратегия | Изменяет данные | Проверяет схему | Проверяет FK | Прерывает на ошибках |
|-----------|----------------|-----------------|--------------|----------------------|
| **STRICT** | ✅ Да | ✅ Строго | ✅ Да | ✅ Да |
| **SAFE_MERGE** | ✅ Да | ⚠️ Гибко | ⚠️ Пропуск | ❌ Нет |
| **FORCE_REPLACE** | ✅ Да | ⚠️ Гибко | ❌ Временно отключены | ❌ Нет |
| **APPEND_ONLY** | ✅ Только новые | ⚠️ Гибко | ⚠️ Пропуск | ❌ Нет |
| **UPSERT_ALL** | ✅ UPSERT операции | ⚠️ Гибко | ❌ Временно отключены | ❌ Нет |
| **DRY_RUN** | ❌ Нет | ✅ Проверка | ❌ Нет | ❌ Нет |
| **SAFE_SCHEMA_CHECK** | ❌ Нет | ✅ Только схема | ❌ Нет | ❌ Нет |

**Legends:**
- ✅ — Полная проверка/изменение
- ⚠️ — Гибкая проверка (допускаются расхождения)
- ❌ — Не проверяется/не применяется

## Архитектура стратегий

```
RestoreStrategy (интерфейс)
    └── AbstractRestoreStrategy (базовый класс)
        ├── StrictRestoreStrategy
        ├── SafeMergeRestoreStrategy
        ├── ForceReplaceRestoreStrategy
        ├── AppendOnlyRestoreStrategy
        ├── UpsertAllRestoreStrategy
        ├── DryRunRestoreStrategy
        └── SafeSchemaCheckRestoreStrategy
```

**Ключевые методы:**
- `performRestore()` — основная логика восстановления (абстрактный)
- `getSql()` — генерация SQL-запросов (абстрактный)
- `handleBatchResult()` — обработка результатов пакетной вставки

**Фабрика стратегий:** `RestoreStrategyFactory.getStrategy(RestoreMode mode)`

**Сервис политик:** `RestorePolicyService.policyFor(RestoreMode mode)`

## Рекомендации по эксплуатации

### 1. Процедура тестирования нового бэкапа

1. **Анализ схемы:** `SAFE_SCHEMA_CHECK` — проверка совместимости
2. **Симуляция:** `DRY_RUN` — предсказание поведения
3. **Реальное восстановление:** Выбор стратегии по сценарию

### 2. Процедура восстановления после сбоя

1. **Оценка потребностей:** Определите, нужно ли полное восстановление или достаточно добавления данных
2. **Выбор стратегии:** 
   - Продакшен с требованиями к точности → `STRICT`
   - Продакшен с приоритетом скорости → `FORCE_REPLACE`
   - Тестовая среда → `DRY_RUN` → `UPSERT_ALL`
3. **Выполнение:** Запустите восстановление с мониторингом

### 3. Регулярное обновление данных

- **Добавление новых записей:** `APPEND_ONLY`
- **Синхронизация данных:** `UPSERT_ALL`
- **Гибкое обновление:** `SAFE_MERGE`

## Заключение

Data Guard предоставляет **гибкую и продуманную систему восстановления** с 7 стратегиями для различных сценариев:

- **Строгие стратегии** (`STRICT`) — для критических сценариев, где нужна гарантия идентичности
- **Гибкие стратегии** (`SAFE_MERGE`, `APPEND_ONLY`) — для постепенного обновления и добавления данных
- **Агрессивные стратегии** (`FORCE_REPLACE`, `UPSERT_ALL`) — для полной замены данных
- **Вспомогательные стратегии** (`DRY_RUN`, `SAFE_SCHEMA_CHECK`) — для предварительной валидации

Выбор стратегии зависит от:
- **Требований к целостности данных**
- **Типа среды** (продакшен/тест)
- **Желаемого поведения при конфликтах**
- **Важности текущих данных** (можно ли их перезаписать)
